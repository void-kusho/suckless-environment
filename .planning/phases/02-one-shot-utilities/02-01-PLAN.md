**battery-notify.c** -- one-shot battery check utility (~80 lines):

Structure:
1. Includes: `<limits.h>`, `<stdio.h>`, `<string.h>`, `<unistd.h>`, `"../common/util.h"`, `"config.h"`
2. Path format macros: `#define CAP_PATH "/sys/class/power_supply/%s/capacity"` and `#define STAT_PATH "/sys/class/power_supply/%s/status"` (per D-05)
3. `static int read_battery(int *cap, char *status, size_t slen)` -- uses snprintf to build paths from BATTERY define, calls pscanf() with `"%d"` for capacity and `"%15[a-zA-Z ]"` for status (handles "Not charging" with spaces per Research Pitfall 6). Returns 0 on success, -1 on failure.
4. `int main(int argc, char *argv[])` -- suppress unused argc with `(void)argc`, set `argv0 = argv[0]`, call `setup_sigchld()`. Call `read_battery()` -- die on failure. Then implement the state logic:
   - If `cap <= THRESHOLD && strcmp(status, "Discharging") == 0`: battery is low and discharging (per BATT-01, BATT-02). Check `access(STATE_FILE, F_OK) != 0` -- if state file does NOT exist, send notification via exec_detach with argv `{"notify-send", "-u", "critical", "Battery Low", body, NULL}` where body is `"Battery at N%"` built with snprintf (per D-07). Then create state file with fopen("w")+fclose (per BATT-03, D-06).
   - Else (not low OR not discharging): call `unlink(STATE_FILE)` to clear state (per BATT-04, D-06). unlink failure (file doesn't exist) is silently ignored (returns -1 with errno ENOENT, which is fine).
5. Return 0.

Code style: K&R braces, tabs for indentation, C99, follow conventions from CLAUDE.md. Use `const char *` for string arrays passed to exec_detach. The `cmd` array for exec_detach must be `const char *const[]` compatible -- declare as `const char *cmd[] = {...}`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| sysfs read | Reading /sys/class/power_supply/BAT1/ pseudo-files (kernel ABI, world-readable) |
| exec boundary | Spawning notify-send via fork+execvp |
| state file | Reading/writing /tmp/battery-notified |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | sysfs path | accept | Battery name comes from compile-time #define BATTERY, not runtime input. No path traversal possible. snprintf with PATH_MAX bounds the output. |
| T-02-02 | Tampering | exec args | mitigate | All argv arrays are const string literals assembled at compile time. No user input flows into exec arguments. capacity integer is formatted with %d into a body string -- no injection vector. |
| T-02-03 | Tampering | state file TOCTOU | accept | access() then fopen() has a tiny race window. Single-user desktop with /tmp state file. Not a real attack surface -- worst case is one duplicate notification. |
| T-02-04 | Denial of Service | state file permissions | accept | If /tmp/battery-notified is not writable (unlikely on standard Linux), fopen fails silently and the next invocation will re-notify. Acceptable degradation. |
| T-02-05 | Information Disclosure | notification content | accept | Battery percentage is not sensitive data. Notification is local-only via dunst. |
</threat_model>
