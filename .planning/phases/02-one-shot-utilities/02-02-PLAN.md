Create utils/screenshot-notify/screenshot-notify.c (~60 lines):

Structure:
1. License comment: `/* See LICENSE file for copyright and license details. */`
2. Includes: `<stdio.h>`, `<stdlib.h>`, `<sys/wait.h>`, `<unistd.h>`, `"../common/util.h"`
3. No config.h -- screenshot-notify has no user-configurable values (per Research assumption A5).

4. `static int capture_to_clipboard(void)` -- two-child pipeline implementing D-08:
   - Create pipe with `pipe(pipefd)`, die on failure.
   - Fork child 1 (maim): close read end, dup2 write end to STDOUT, close write end, `execlp("maim", "maim", "-s", NULL)`, `_exit(127)` on exec failure.
   - Fork child 2 (xclip): close write end, dup2 read end to STDIN, close read end, `execlp("xclip", "xclip", "-selection", "clipboard", "-t", "image/png", NULL)`, `_exit(127)` on exec failure.
   - Parent: close BOTH pipe ends immediately (per Research Pitfall 5 -- prevents xclip hang).
   - `waitpid(maim_pid, &status, 0)` -- wait for maim to get exit code.
   - `waitpid(xclip_pid, NULL, 0)` -- wait for xclip to finish.
   - Return 0 if `WIFEXITED(status) && WEXITSTATUS(status) == 0`, else return 1 (per D-09).

5. `int main(int argc, char *argv[])`:
   - Suppress unused argc with `(void)argc`.
   - Set `argv0 = argv[0]`.
   - Call `setup_sigchld()` (per Research Pitfall 3 -- zombie prevention).
   - Call `capture_to_clipboard()`. If returns 0 (success):
     - Send notification via exec_detach with argv `{"notify-send", "Screenshot", "Image copied to clipboard", NULL}` (per D-10). No -u flag -- use default "normal" urgency (per Research open question 1, Claude's discretion).
   - If returns non-zero (cancel or failure): do nothing, just fall through (per D-11 -- silent exit).
   - Return 0 always (per D-11 -- exit 0 even on cancel).

Code style: K&R braces, tabs for indentation, C99. No shell invocation anywhere -- pure fork+exec.

CRITICAL: In capture_to_clipboard, each child process MUST close the pipe end it does not use BEFORE calling dup2, and MUST close the dup2'd end AFTER dup2 (since dup2 does not close the original). The correct sequence for child 1 is: close(pipefd[0]), dup2(pipefd[1], STDOUT_FILENO), close(pipefd[1]). For child 2: close(pipefd[1]), dup2(pipefd[0], STDIN_FILENO), close(pipefd[0]).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| exec boundary (maim) | Spawning maim via fork+execlp -- maim interacts with X11 for area selection |
| exec boundary (xclip) | Spawning xclip via fork+execlp -- xclip receives piped PNG data |
| exec boundary (notify-send) | Spawning notify-send via exec_detach -- fire-and-forget notification |
| pipe data flow | PNG image data flows from maim stdout through pipe to xclip stdin |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06 | Tampering | exec args | mitigate | All argv arrays are const string literals. No user input flows into exec arguments. Args for maim ("-s"), xclip ("-selection", "clipboard", "-t", "image/png"), and notify-send are hardcoded at compile time. |
| T-02-07 | Denial of Service | pipe deadlock | mitigate | Parent closes both pipe ends immediately after forking both children (per Research Pitfall 5). This ensures xclip receives EOF when maim finishes, preventing indefinite hang. |
| T-02-08 | Denial of Service | zombie processes | mitigate | setup_sigchld() installs SIGCHLD handler that reaps zombies via waitpid(-1, NULL, WNOHANG). Additionally, parent explicitly waits for both children with waitpid(). |
| T-02-09 | Tampering | pipe data | accept | PNG data flows through an in-process pipe between two child processes. No external network involved. An attacker with local access could theoretically intercept pipe data, but that requires root or same-user access which already grants clipboard access anyway. |
</threat_model>
